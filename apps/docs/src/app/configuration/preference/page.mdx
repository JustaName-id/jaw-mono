# preference

Advanced configuration options to customize SDK behavior and authentication mode.

**Type:** `object`
**Required:** No
**Default:** See individual options below

---

## Usage

```typescript
import {JAW, Mode} from '@jaw.id/core';

const jaw = JAW.create({
  apiKey: 'your-api-key',
  preference: {
    mode: Mode.AppSpecific,
    showTestnets: true,
    ens: 'myapp.eth',
  },
});
```

---

## Options

### mode

Authentication mode that determines where passkey operations occur.

**Type:** `Mode.CrossPlatform` | `Mode.AppSpecific`
**Default:** `Mode.CrossPlatform`

#### Mode Comparison

| Feature | Mode.CrossPlatform | Mode.AppSpecific |
|---------|---------------|-------------|
| **Passkey operations** | On `keys.jaw.id` (popup) | Within your dApp |
| **User experience** | Redirects to popup | Stays in your app |
| **Wallet reuse** | ✅ Universal | ❌ App-specific only |
| **Origin binding** | `keys.jaw.id` | Your domain |
| **Branding** | JAW interface | Can use custom UI |

<Callout type="warning">
**Important:** Passkeys are always stored by the browser/device (not on any server). The mode determines where passkey operations happen and whether wallets can be reused across applications.
</Callout>

#### CrossPlatform Mode

Passkey operations redirect to `keys.jaw.id` in a popup, enabling wallet reuse across multiple applications.

```typescript
import {JAW, Mode} from '@jaw.id/core';

const jaw = JAW.create({
  apiKey: 'your-api-key',
  preference: {
    mode: Mode.CrossPlatform,
  },
});
```

**How it works:**
- User clicks "Connect" → Popup opens to `keys.jaw.id`
- Passkey operations happen on `keys.jaw.id` origin
- Wallet can be reused across any app using CrossPlatform mode
- Users see consistent experience across applications

**Benefits:**
- Single wallet across multiple dApps
- Seamless cross-application experience
- Users don't recreate wallets for each app
- Consistent authentication flow

#### AppSpecific Mode

Passkey operations stay within your dApp (no redirect). Wallets are specific to your application's origin.

```typescript
import {JAW, Mode} from '@jaw.id/core';

const jaw = JAW.create({
  apiKey: 'your-api-key',
  preference: {
    mode: Mode.AppSpecific,
  },
});
```

**How it works:**
- User clicks "Connect" → Modal opens within your dApp
- Passkey operations happen on your dApp's origin
- Passkeys are tied to your domain
- Users never leave your application

**Benefits:**
- Users never leave your dApp
- Full control over UI/UX
- White-label experience
- No external redirects

### serverUrl

URL to a passkey metadata server for managing passkey credentials (credential IDs, public keys, display names).

**Type:** `string`
**Default:** `'https://api.justaname.id/wallet/v2/passkeys'`
**Only works with:** `Mode.AppSpecific`

```typescript
import {JAW, Mode} from '@jaw.id/core';

const jaw = JAW.create({
  apiKey: 'your-api-key',
  preference: {
    mode: Mode.AppSpecific,
    serverUrl: 'https://custom-backend.example.com/passkeys',
  },
});
```

<Callout type="warning">
**Important:** This is NOT for storing passkeys themselves (passkeys are always stored by the browser/device). The server stores metadata about passkeys to enable passkey lookups and management.
</Callout>

**When to Customize:**
- Self-hosted passkey metadata server
- Development/staging environments
- Custom passkey management logic

#### Running Your Own Passkey Server

If you wish to run your own passkey server, it must implement the following endpoints:

**1. GET / - Lookup Passkeys by Credential IDs**

Retrieves passkey information for one or more credential IDs.

Query Parameters:
- `credentialIds` (string[], repeatable): Array of credential IDs to lookup

Response Format (200):
```json
{
  "statusCode": 200,
  "result": {
    "data": {
      "passkeys": [
        {
          "credentialId": "string",
          "publicKey": "0x...",
          "displayName": "string"
        }
      ]
    },
    "error": null
  }
}
```

Error Response (404):
```json
{
  "statusCode": 404,
  "result": {
    "data": null,
    "error": "Passkeys not found"
  }
}
```

**2. POST / - Register a New Passkey**

Registers a new passkey credential.

Request Body:
```json
{
  "credentialId": "string",
  "publicKey": "0x...",
  "displayName": "string"
}
```

Headers:
- `Content-Type: application/json`

Response:
- Status `200`/`201` for successful registration
- Status `4xx`/`5xx` for errors

**Data Format Requirements:**
- **Public Keys**: Must be hex-encoded strings prefixed with `0x`
- **Credential IDs**: Base64url-encoded strings (standard WebAuthn format)
- **Display Names**: Human-readable names for the passkeys

### uiHandler

Custom UI handler for rendering approval dialogs in app-specific mode. When provided, the SDK will use this handler instead of redirecting to external popups.

**Type:** `UIHandler`
**Default:** `undefined`
**Only works with:** `Mode.AppSpecific`

<Callout type="info">
For React applications, use `JAWUIProvider` from `@jaw/ui` which provides a pre-built UIHandler implementation. See the [UI Package documentation](/ui-package) for details.
</Callout>

#### Using @jaw/ui (Recommended for React)

```typescript
import { JAWUIProvider } from '@jaw/ui';

function App() {
  return (
    <JAWUIProvider
      apiKey="your-api-key"
      appName="My DApp"
      defaultChainId={1}
      paymasterUrls={{ 1: 'https://...' }}
    >
      <YourApp />
    </JAWUIProvider>
  );
}
```

#### Custom UIHandler Implementation

For non-React apps or custom UI, implement the `UIHandler` interface:

```typescript
import {
  JAW,
  Mode,
  UIHandler,
  UIRequest,
  UIResponse
} from '@jaw.id/core';

const customUIHandler: UIHandler = {
  async request<T>(request: UIRequest): Promise<UIResponse<T>> {
    // Render your custom UI based on request.type
    switch (request.type) {
      case 'wallet_connect':
        // Show connection dialog
        break;
      case 'personal_sign':
        // Show signature dialog
        break;
      case 'eth_signTypedData_v4':
        // Show EIP-712 dialog
        break;
      case 'wallet_sendCalls':
        // Show transaction dialog
        break;
      case 'wallet_grantPermissions':
        // Show permission grant dialog
        break;
      case 'wallet_revokePermissions':
        // Show permission revoke dialog
        break;
      case 'wallet_sign':
        // Handle unified signing (0x45 or 0x01)
        break;
    }

    return {
      id: request.id,
      approved: true, // or false if rejected
      data: result,
    };
  },

  canHandle(request: UIRequest): boolean {
    return true; // Return true for supported request types
  },

  async cleanup(): Promise<void> {
    // Clean up any open dialogs
  },
};

const jaw = JAW.create({
  apiKey: 'your-api-key',
  preference: {
    mode: Mode.AppSpecific,
    uiHandler: customUIHandler,
  },
});
```

#### UIRequest Types

| Type | Description | Data Fields |
|------|-------------|-------------|
| `wallet_connect` | Connection request | `appName`, `appLogoUrl`, `origin`, `chainId`, `capabilities` |
| `personal_sign` | EIP-191 signature | `message`, `address` |
| `eth_signTypedData_v4` | EIP-712 signature | `typedData` (JSON), `address` |
| `wallet_sendCalls` | Transaction batch | `version`, `from`, `calls[]`, `chainId` |
| `wallet_grantPermissions` | Permission grant | `address`, `chainId`, `expiry`, `spender`, `permissions` |
| `wallet_revokePermissions` | Permission revoke | `permissionId`, `address`, `chainId` |
| `wallet_sign` | Unified signing | `address`, `chainId`, `request: { type, data }` |

#### UIResponse Structure

```typescript
interface UIResponse<T> {
  id: string;        // Must match request.id
  approved: boolean; // User approved or rejected
  data?: T;          // Result data (signature, tx hash, etc.)
  error?: UIError;   // Error if rejected
}
```
